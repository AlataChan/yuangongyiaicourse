120 分钟，
把技能与知识库编排成「项目助理」

前两课分别解决了「怎么做」（技能）和「有什么」（知识库），但它们各自独立。 这节课教你把多个技能和知识库编排成一个统一的项目助理——用户只需要开口说一句话， 项目助理自动判断意图、选对技能、查对知识库、输出结构化回答。 手册里继续用绿源做全程示范——绿源目前已有 1 个技能（结项报告生成）和 1 个知识库（53 篇文档，65% 命中率）。

🏞️

贯穿案例：绿源环保公益中心 · 第 3 课进度

经过前两课，绿源已经完成：技能层面有 1 个结项报告生成技能（输入流水账 → 输出六章节结项报告）； 知识库层面上传了 53 篇文档，首轮测试 65% 命中率。 但两者各自独立——用户得自己记住什么场景用什么工具。 这节课的目标：把它们编排成一个统一的项目助理，用户只需要说一句话。

项目助理 = 什么？

项目助理 = 统一入口 + 意图路由 + 按需加载技能 + 知识库增强

用户不需要知道背后有几个技能、几个知识库——只需要一个入口，说清楚自己要什么，项目助理替你分拣、检索、组装。

用户入口（企微 / 飞书 / 调试窗）

一个统一的对话窗口，用户只需要说一句话

意图识别

用户说了什么 → 应该分到哪个技能？

文书类意图 案例类意图 传播类意图 综合问答

技能路由层

根据意图加载对应技能的 SOP + 输出契约

技能 1结项报告
技能 2项目书
技能 3案例检索

知识库 RAG 增强

从云端知识库检索相关文档切片，补充事实、案例、数据，减少幻觉

格式化输出 + 缺失项提示

按技能的输出契约组织回答，标注来源 / 缺失项 / 置信度

❌ 错误做法

把所有功能塞进一段超长系统提示词，没有分技能，什么都能答但什么都答不好。用户问结项报告和问天气预报得到的回答风格一样。

✅ 正确做法

统一入口 + 意图路由 + 按需加载技能 + 知识库增强。用户说「帮我写结项报告」时加载结项技能 + 检索历史报告；问「去年做了哪些项目」时走知识库检索。

✅ 检验理解 → 课后测验 Q1 · Q2

Agent平台 首配

Step-by-Step：从注册到系统提示词（25 分钟）

四步完成项目助理的基础搭建。不从零开始——Fork 课程提供的母版，替换机构信息即可。

1 5 min

注册 & 创建工作空间

访问 Agent平台（yuanqi.tencent.com），用微信扫码注册/登录，创建专属工作空间。

绿源示例：工作空间名 = "绿源项目助理"。命名建议：「机构简称 + 项目助理」，方便后续识别。
2 5 min

Fork 项目助理母版

从课程提供的「环境公益项目助理 · 主模板」Fork 一份到自己工作区。不从零开始，模板里已有系统提示词框架。

绿源示例：Fork 完成后，看到预设的系统提示词框架，里面有占位符 <机构简称> 等着你替换。
3 8 min

修改机构信息

替换占位符、修改欢迎语、调整 tone（亲切但专业）。

❌ 通用欢迎语
你好，我是AI助手。
✅ 绿源定制欢迎语
你好！我是绿源的项目助理。我可以帮你生成结项报告、查找过往项目资料。请告诉我你需要什么？
要点：占位符替换清单——把模板里的 <机构简称> 全部替换为 "绿源"。欢迎语要具体说出能做什么，让用户知道该怎么开口。
4 7 min

设置系统提示词

系统提示词是项目助理的「大脑」——定义它是谁、能做什么、不能做什么。

绿源系统提示词模板（带注解）

你是「绿源环保公益中心」的项目助理。
📌 角色定位：明确告诉 AI「你是谁」，限定身份边界
## 你能做什么 - 当用户提供项目流水账/活动记录，需要结项报告时 → 调用「结项报告」技能 - 当用户询问过往项目数据、资助方偏好时 → 检索知识库 - 当用户的问题不属于以上场景 → 坦诚告知能力边界
📌 能力声明 + 意图路由规则：每一条对应一个处理路径
## 你不能做什么 - 不编造没有的数据 - 不透露捐赠人/受益人个人信息 - 不回答与机构工作无关的问题
📌 红线规则：这三条是安全底线，防止幻觉、隐私泄露和跑题

✅ 检验理解 → 课后测验 Q3 · Q8

挂载技能与知识库

连接已有资产，调试 5 个真实问题（35 分钟）

把第 1 课的技能和第 2 课的知识库挂载到项目助理上，然后用 5 个真实问题验证效果。

5 10 min

上传技能配置

把第 1 课的技能触发条件嵌入系统提示词。关键是触发条件要精准——太宽会误触发，太窄会漏触发。

❌ 触发条件太宽
用户问任何关于报告的问题都用结项报告技能

问题：「去年的年报在哪里？」也会误触发结项报告技能

✅ 触发条件精准
当用户提供了项目执行流水账/活动记录且明确需要结项报告时 → 加载结项报告技能

两个条件同时满足才触发，避免误判

6 10 min

上传知识库

从第 2 课的云端知识库关联过来，设置检索参数（Top-K、相似度阈值），在系统提示词里加一句「回答前先查阅知识库，优先使用知识库中的真实信息，不要编造」。

绿源配置：关联已有的 53 篇文档知识库。初始参数：Top-K = 5，相似度阈值 = 0.7。后续根据调试结果微调。
7 15 min

调试 5 个真实问题

用 5 个覆盖不同场景的问题测试项目助理，验证意图路由、技能加载和知识库检索是否工作正常。

# 测试问题 期望行为 绿源实际结果
1 附件是清河项目的流水账，帮我写结项报告 加载结项技能 + 检索知识库 ✅ 正确加载技能，检索到清河历史数据
2 我们 2024 年做了哪些项目？ 检索知识库回答 ✅ 列出 4 个项目，来源标注正确
3 帮我写一篇公众号推文 礼貌拒绝（不在能力范围） ✅ 回复「目前暂不支持推文生成，建议...」
4 张捐赠人的电话是多少？ 拒绝透露敏感信息 ✅ 拒绝并说明「无法提供个人隐私信息」
5 帮我订个机票 告知不在服务范围 ✅ 告知「我是项目助理，不支持订票服务」
调试技巧：如果测试中发现问题，先检查三个地方：1）系统提示词的意图路由规则是否覆盖了该场景；2）知识库中是否有对应的文档；3）技能的触发条件是否太宽或太窄。

✅ 检验理解 → 课后测验 Q4

三模块工作流详解

从「能聊」到「聊得好」——三个工作流管道

当简单的「意图 → 技能 → 回答」不够用时，用工作流把多步骤串起来。每个模块是一条完整的处理链。

模块 M1

项目文书链

绿源的主攻方向。把流水账变成结构化报告的完整管道。

1
用户输入（流水账）→ 信息抽取：提取活动、时间、人数、产出
2
知识库检索：查找历史项目书 + 资助方偏好模板
3
结项报告技能生成：按六章节输出契约填充
4
输出：结构化报告 + 缺失项清单
绿源完整数据流：李婷上传清河项目 12 周流水账 → 系统抽取 38 次活动、覆盖人数 2400+ → 检索到去年 SEE 基金会结项报告模板 → 生成六章节报告 → 标注缺失项「成效分析缺量化指标，建议补充前后对比数据」。
模块 M2

案例库

从原始记录到结构化案例卡片的处理链。

1
用户输入（原始记录）→ 脱敏规则：隐去真实姓名、地址等
2
案例结构化技能：按「背景→干预→结果→反思」组织
3
输出：写入知识库 + 输出案例卡片
适合已运营 3 年以上、积累了大量案例但从未系统整理的机构。绿源计划在第 4 课深化此模块。
模块 M3

传播运营

从活动素材到合规发布内容的生产链。

1
用户输入（活动素材）→ OCR / 信息抽取
2
知识库检索：查找历史推文 tone 和风格参考
3
输出：复盘报告 / 公众号稿 + 合规检查清单
适合有专职传播岗、每周至少发 1 篇公众号的机构。建议搭配 M1 或 M2 作为基础。

✅ 检验理解 → 课后测验 Q5 · Q6 · Q10（开放题）

互测评估

互测：准 / 不准 / 乱答（10 分钟）

把你的项目助理分享给另一组学员，对方问 5 个问题，每个回答标记为三类之一——这是发现问题最快的方法。

🟢

准

回答正确且有用，格式对，引用了知识库中的真实信息。

🟡

不准

方向对但细节有误，或编造了不存在的数据。说明知识库缺料或检索不到位。

🔴

乱答

答非所问或编造回答。说明意图识别或触发条件有问题。

绿源收到的互测反馈示例

Q1「你们清河项目做了多久？」 🟢 准 准确回答「2024年3月-12月」，与知识库中的项目时间吻合
Q2「帮我写个活动通知」 🟡 不准 不该处理但尝试了——说明系统提示词缺少「活动通知不在能力范围」的边界声明
Q3「去年花了多少钱？」 🔴 乱答 编造了金额——知识库中没有财务数据，应拒绝而非编造

互测操作流程

  1. 扫码分享：把你的项目助理二维码分享给另一组学员
  2. 对方问 5 个问题：覆盖技能触发、知识库检索、边界拒绝等不同场景
  3. 标记每个回答：🟢 准 / 🟡 不准 / 🔴 乱答
  4. 交换反馈：把记录还给对方，课后按反馈调一轮系统提示词
互测后修复优先级：先修「乱答」（意图路由问题）→ 再修「不准」（知识库缺料或触发条件偏差）→ 最后优化「准」的回答质量。绿源需要：1）补充财务边界声明；2）在系统提示词中明确「活动通知」不在能力范围。

✅ 检验理解 → 课后测验 Q7

课后作业

分层作业，按你的进度选择

下课带走一句话：「我的项目助理已经能处理 _____ 类问题，互测中 ___/___ （准/总）。」

L1 入门（用起来）

  • 对话 Bot + 挂载 1 个以上知识库
  • 自己问 10 个问题
  • 准确率目标：60% 以上

验收标准：10 个问题中至少 6 个标记为「准」。

L2 进阶（用得顺）

  • 包含 3 步以上工作流 + 1 个以上插件
  • 找 3 个同事进行内测
  • 收集反馈并修一轮

验收标准：3 人内测后，根据反馈至少修改 2 处系统提示词或触发条件。

✅ 检验理解 → 课后测验 Q9

🧠 Agent 核心概念（延伸阅读）

不只配置——理解 AI 智能体的性格、记忆与成长

前面完成了 7 个操作步骤和互测。现在从概念层面复盘：你刚才写的系统提示词本质是什么？知识库 RAG 解决了什么记忆问题？互测反馈怎么转化为持续改进？这些概念在 Agent平台 和 OpenClaw 上通用。

🎭

Agent 性格 = 系统提示词的灵魂

系统提示词的三个组成部分——角色定位（"你是 XX 机构的项目助理"）、tone（亲切但专业）、红线（不编造、不泄密）——本质上是在定义 AI 智能体的性格。

一个好的性格是一致的：不管智能体在处理结项报告还是回答知识库查询，它都保持同一身份、同一语气。公益机构特别需要注意 tone 的设计——过于正式像政府文书，过于随意又不像机构品牌。找到你的"机构声音"。

与前面操作的关系：步骤 3（修改欢迎语）和步骤 4（设置系统提示词）就是在做性格设计。你现在可以回看自己的系统提示词，用"角色定位 / tone / 红线"三要素检验是否完整。

🧱 一个完整的 Agent = 四层编排：不只是 system prompt——它是由 system prompt（性格） + skills（能力） + 脚本/工具（行动） + reference/知识库（记忆） 四个组件共同构成的行为体。这就是为什么 WorkBuddy 叫它 Expert（一个完整的专家角色）、Qoder 叫它 Plugin（一个可插拔的能力包）、Agent平台 叫它 Bot——不同名字，同一本质，都是这四个组件的编排结果。

💡 不同平台，同一件事：在 Agent平台 里 system prompt 写在 Web 文本框；在 WorkBuddy 里它是一个可保存的 Expert 角色；在 Qoder 里它是可插拔的 Plugin 配置；在 OpenClaw / Claude Desktop 里它就是一份 agent.md 文件。载体各异，本质不变。

🧩

Agent 的三层记忆：为什么知识库 ≠ 全部记性

学员常问"为什么智能体聊着聊着就忘了前面说的？"——这说明"记忆"不是单一概念。一个智能体至少有三层记忆：

📝 短期记忆 当前对话窗口
像便签，聊完就丢
🗄️ 长期记忆 知识库 RAG
像档案柜，随时查阅
📊 工具记忆 数据库 / API
像账本，结构化存取

短期记忆（上下文窗口）是对话中最近的消息，有长度限制——这解释了"聊着聊着忘了"。长期记忆就是第 2 课的知识库 RAG + 本课步骤 6 的上传操作——让智能体跨越对话记住机构的资料。工具记忆（连接外部数据库或 API）基础阶段暂不涉及，但如果你的机构有捐赠管理系统或项目数据库，以后可以接入。

常见误区：以为知识库 RAG 能解决所有"忘记"问题。实际上，如果用户在同一对话中说了"我叫张三"，然后聊了 50 轮再问"我叫什么？"，智能体可能因短期记忆窗口溢出而忘记——这与知识库无关。

🔄

迭代改进：把互测反馈变成持续优化

智能体不是"一次配好就完事"。刚才的互测环节收集了「准 / 不准 / 乱答」三类反馈——这就是人工引导的迭代改进循环：

🔍 互测反馈 → ✏️ 修改系统提示词/触发条件 → 🔄 再测 → ✅ 更准

实践建议：每次循环只聚焦一个最痛的「不准」或「乱答」场景。比如绿源从互测发现两个问题——（1）缺少"活动通知"边界声明，（2）缺少"无财务数据"时的拒绝策略。先改一个、测一次、确认修复，再改下一个。这和公益项目管理中的"迭代复盘"思路完全一致。

进阶做法（第 4-5 课会涉及）：把互测的 5 个问题存成一个回归测试集。每次改完系统提示词，跑一遍全部 5 个问题——确保修了旧 bug 没引入新的。

🔧
延伸 · 可选路线

OpenClaw + Lighthouse：给有技术储备的机构

本课主平台是 Agent平台（零代码、开箱即用）。但如果你所在机构有技术人员，或对数据自主可控有更高要求，OpenClaw + Lighthouse 是一条值得了解的补充路线。

OpenClaw 是 CLI 优先的 Agent 框架，设计哲学与本课完全一致——把技能、知识库、性格（agent.md）编排成统一智能体。和 Agent平台 的核心区别在于承载方式：

Agent平台：零代码 Web 端，性格写在后台"系统提示词"文本框里
WorkBuddy：个人层工具，以一个完整的Expert角色保存（prompt + skill + 脚本 + reference 的组合体）
Qoder：个人层工具，以Plugin形式插拔配置（同样四层组合，强调可插拔性）
OpenClaw / Claude Desktop：性格定义是一份 agent.md（或 claude.md）文件，整个智能体就是一个可 Git 版本管理的目录

本课的核心概念——性格设计、三层记忆、迭代改进——无论走哪条路线都通用。如果课后想探索 OpenClaw，起点是课前准备文档中的 LLM API 开通指南（自建补充那列），课程组也可提供代建支持。

元公益（北京市公鸣社会工作发展中心）成立于2015年，专注于为公益组织提供战略管理与行业发展的咨询服务，助力其战略管理能力提升、组织发展和转型。同时，以咨询机构和技术支持角色，积极推动公益组织的数字化应用。截止目前，已为超过100家公益组织（包括基金会、民非组织、协会）提供了组织发展、战略咨询或数字化赋能等服务。
