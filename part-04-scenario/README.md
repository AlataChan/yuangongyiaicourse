## 一、这节课要解决什么

第 3 课让项目助理「跑起来」了，但「什么都能聊一点」≠「什么都好用」。大多数机构的人力只够把 1-2 个场景打磨到真正替代手工——与其面面俱到但都半成品，不如**选你机构最疼的那条链路，纵深打磨**。

第 4 课做两件事：

1. **按模块分流深化**——三个模块各有不同的打磨方向，你跟着自己的模块走。
2. **补齐第 2 个技能**——让项目助理从「只会一招」变成「有两个拿手菜」，同时保证两个技能不打架。

一句话：**把「能用」拉到「好用」，关键不是加功能，而是把你的主场景打磨到位、再多会一个技能。**

## 二、三个模块的深化方向

不同模块的痛点不同，深化重点也不同。找到你在第 0 课选定的模块，对号入座：

### M1 文书链

| 深化方向 | 具体动作 |
|---------|---------|
| 立项→结项链路串联 | 把项目书技能的输出字段（目标、指标、预算）映射为结项报告技能的输入检查项——两份文书数据对齐 |
| 第二技能推荐 | **资助信 / 项目摘要**：把项目核心信息浓缩成一页纸申请信，和项目书技能形成「长文书 + 短文书」配合 |
| 打磨要点 | 输出契约：结项报告六章节 vs. 资助信一页纸，两个技能的输出长度、章节完全不同，不会误触发 |

### M2 案例库

| 深化方向 | 具体动作 |
|---------|---------|
| 检索命中率优化 | 调整 Top-K、相似度阈值、文档切片粒度，用 5 个典型查询测覆盖率和精准度 |
| 第二技能推荐 | **案例萃取 / 脱敏**：把一段原始项目记录萃取成「背景→干预→结果→反思」四段式案例卡，同时自动脱敏 |
| 打磨要点 | 案例检索（查已有案例）和案例萃取（从原始材料生产新案例）触发边界清晰：「找案例」→ 检索，「写案例」→ 萃取 |

### M3 传播运营

| 深化方向 | 具体动作 |
|---------|---------|
| tone 一致性 | 用 3 篇机构历史推文提取语气基准，写进技能的 Output Contract，确保 AI 输出的语气像「我们机构的人写的」 |
| 第二技能推荐 | **活动复盘 / 社群 FAQ**：把活动执行细节转成社群可发的 FAQ 或复盘帖，和推文技能形成「对外传播 + 对内沉淀」配合 |
| 打磨要点 | 推文技能（对外、需要感染力）和 FAQ 技能（对内、需要准确简洁）语气差异大，触发说明要写清楚 |

## 三、构建你的第二个技能

### 3.1 从立项书出发（15 分钟）

回到第 1 课的立项书，找到「Skill 2」那一栏——你当时已经预留了第二技能的场景。现在正式构建它：

```text
构建步骤：
- [ ] Step 1  确认场景：从上面模块推荐中选一个，或用你立项书里的 Skill 2
- [ ] Step 2  写触发说明：一句「什么情况下用我」+ 明确「不该用我」的场景
- [ ] Step 3  写输出契约：固定输出结构、字段、字数范围
- [ ] Step 4  写 6 条 Gotchas：最容易翻车的 6 点（参照第 1 课方法）
- [ ] Step 5  在 QoderWork 里跑通一遍
```

### 3.2 路由不冲突自检（20 分钟）

两个技能挂在同一个项目助理里，最常见的问题是**路由冲突**——用户说一句话，两个技能都觉得该自己接。

**技能路由不冲突自检模板**：

| 检查项 | 技能 1 | 技能 2 | 是否冲突 |
|--------|--------|--------|---------|
| 触发关键词 | 例：结项报告、项目总结 | 例：资助信、申请信 | 有无重叠？ |
| 触发意图描述 | 「用户需要结构化结项报告时」 | 「用户需要一页纸申请信时」 | 能否一句话区分？ |
| 输出格式 | 六章节长报告 | 一页纸信件 | 格式差异够大？ |
| 容易混淆的用户说法 | 「帮我写个总结」 | 「帮我写个简介」 | 列出 3 条易混说法并明确归属 |
| 兜底规则 | 如果不确定 → 问用户「你要完整报告还是一页纸摘要？」 | 同左 | 兜底是否一致？ |

**三版触发说明改写（复习第 1 课方法）**：

1. 功能版（不推荐）：`帮公益机构写申请信。`
2. 意图版：`当用户需要一封简短的资助申请信时使用。`
3. 带边界版（提交版）：`当用户提供项目核心信息、需要一页纸资助申请信或项目摘要时加载；不要用于完整项目书或结项报告——那些走别的技能。`

> 关键原则：两个技能的 description 放在一起读，应该像两个岗位的 JD——各管各的、边界清晰、有交叉时明确「归谁」。

## 四、模块化互测

### 4.1 分组（2 分钟）

同模块的学员 2-3 人一组。

### 4.2 交叉测试（10 分钟）

每人用对方的项目助理，各问 3 个问题（至少覆盖两个技能场景），记录反馈：

| 用户说 | 期望加载的技能 | 实际表现 | 分类 |
|--------|-------------|---------|------|
| | 技能 1 / 技能 2 | | 准 / 不准 / 乱答 |
| | | | |
| | | | |

**分类标准**：

- **准**：加载了对的技能、输出格式正确、内容基于真实信息
- **不准**：加载了对的技能，但内容有误或编造了数据
- **乱答**：加载了错误的技能、两个技能打架、或答非所问

### 4.3 修一轮（3 分钟）

根据互测反馈，当场修改最紧急的 1 个问题（通常是触发说明的边界不够清晰）。
